\section{Evaluation}

The mechanisms introduced in \S\ref{s:design}--\S\ref{s:gossip}---dual-recording, blind validation, and gossip dissemination---add overhead relative to standard public transactions. We evaluate this cost by answering: (1) What is the baseline end-to-end overhead of privacy? (2) How sensitive is performance to dissemination parameters ($n_{dissem}$, $n_{required}$)? (3) How does the system scale with collection count and network size? and (4) How resilient is the system to churn and network faults?

\noindent\textbf{Experimental Setup.}
We design four application workloads modeling multi-party business processes to stress different dimensions of the PDC mechanism (Table~\ref{table:workloads}). Supply Chain maximizes the collection count per organization; Auction isolates dissemination throughput; Trade Finance stresses scalability under strict majority endorsement; and EHR tests read throughput. For each, we compare a \textit{PDC variant} against an \textit{all-public variant} to isolate privacy overhead. In addition, we develop \texttt{PDC\_BENCH}, a minimal chaincode that exposes four operations---\texttt{WritePrivateData}, \texttt{ReadPrivateData}, \texttt{WritePublicData}, and \texttt{ReadPublicData}---enabling controlled measurement of individual PDC mechanism overhead against a public-ledger baseline.

\noindent\textbf{Commit-Path Overhead and Optimizations.}
Initial microbenchmarks of the commit path using \texttt{PDC\_BENCH} (4 organizations, 4 peers) revealed a 50\% throughput penalty for PDCs (720 vs 1440 tx/s). Profiling showed the gap was dominated by cryptographic membership checks and serial disk I/O. We applied a series of crash-safe optimizations (Table~\ref{table:optimizations}): replacing cryptographic checks with O(1) identifier comparisons, parallelizing block and private data stores, deferring purges to background routines, and adding a transient-store cache. Together, these reduced commit-path overhead to just 6\% relative to public-only transactions.

\subsection{End-to-End Performance}
To answer EQ1, we compare the PDC and all-public variants of all four workloads at peak throughput. In each PDC variant, roughly 40--50\% of transactions touch private data collections. Figure~\ref{fig:e2e-throughput} visualizes throughput and commit latency side by side.
Across all four workloads, introducing PDCs consistently reduces peak throughput: 3\% for Auction, 7.5\% for Supply Chain, 10\% for Trade Finance, and 14\% for EHR (Figure~\ref{fig:e2e-throughput}). The overhead correlates with per-transaction complexity: EHR is the most read-heavy workload (3 reads per tx), and each private-data read requires an additional transient-store or private-data-store lookup, amplifying the cost. Trade Finance suffers from its strict 7-of-12 endorsement policy, which produces the highest baseline endorsement latency; the additional dissemination and commit-path work compounds an already long critical path. Auction and Supply Chain, with simpler read-write profiles and lighter endorsement policies, see the smallest overheads.
Commit latency increases uniformly by 4--17\% for the first three workloads, but jumps by 40\% for Trade Finance. Trade Finance's outsized latency increase reflects the interaction between its heavy endorsement phase and the PDC commit path---the committer must verify private-data hashes and potentially trigger reactive retrieval on top of an already slow endorsement round.

\textit{Where does the overhead originate?} (EQ2). To decompose the cost, we collected per-node Prometheus metrics and disk I/O counters during each 300\,s run. Three sources account for the gap. First, per-transaction CPU cost rises by 10--20\% due to hash computation and verification, transient-store writes, and private-data-store flushes---none of which exist in the all-public path. Second, because the ordering service never sees private payloads, they must be distributed point-to-point via the gossip protocol, increasing per-transaction network traffic by 10--15\%; missed payloads trigger reactive retrieval that further inflates peer-to-peer traffic. Third, disk reads increase 2--4$\times$ as committers query the transient store and private-data store to match payloads against on-chain hashes. Conversely, total disk writes \textit{decrease}: in the all-public path every peer writes the full payload, whereas with PDCs the public ledger records only a hash and only authorized peers persist the private payload---a storage efficiency benefit of selective replication.

\subsection{Sensitivity and Scalability}
\textit{Dissemination Parameter Sensitivity} (EQ3).
To isolate the effect of the dissemination parameters introduced in Algorithm~\ref{alg:dissemination}, we deploy a 20-organization network (one peer per organization) across four WAN regions (US East, EU Germany, Japan, US South; five peers per region), with three Raft orderers co-located in US East and three client machines generating load. All peers share a single private data collection spanning all 20 organizations under an OR membership policy. The workload consists entirely of private-data write transactions (100-byte values), each requiring a single endorsement, running at a target load of 600\,tx/s for 300\,s.

We study two dissemination strategy families by varying the collection configuration parameters $n_{dissem}$ and $n_{required}$:
\begin{itemize}[topsep=0pt,noitemsep,leftmargin=15pt]
\item \textit{Pull-dominant:} The endorsing peer proactively pushes to $n_{dissem} \in \{0, 4, 9, 14, 19\}$ peers before ordering; the remaining $20 - n_{dissem}$ peers retrieve private data reactively at commit time ($n_{required} = 0$, so endorsement latency is unaffected by dissemination).
\item \textit{Push-with-quorum:} The endorsing peer pushes to all 20 peers ($n_{dissem} = 20$) but blocks the endorsement response until $n_{required} \in \{0, 4, 9, 14, 19\}$ peers acknowledge receipt.
\end{itemize}

Figure~\ref{fig:dissemination} shows performance under pull-dominant vs. push-with-quorum strategies. Relying on commit-time pulls (panels a, c) creates catastrophic bottlenecks: if $n_{dissem}$ is too low, simultaneous pull requests trigger timeouts, collapsing throughput and causing data loss. Conversely, blocking endorsements on WAN acknowledgments (panels b, d) preserves data but creates backpressure that throttles throughput (dropping 81\% at $n_{required}=19$). We conclude that asynchronous push ($n_{dissem}=N-1, n_{required}=0$) is the optimal WAN configuration: it guarantees retention, maximizes throughput, and eliminates commit-time pull storms.

\textit{Scalability with Collections and Network Size} (EQ4).
To understand how the system scales when both network size and collection count grow simultaneously, we increase the number of organizations from 4 to 20 (in increments of 4) and define a bilateral private data collection for every pair of organizations, yielding $\binom{n}{2}$ collections: 6, 28, 66, 120, and 190 collections for $n = 4, 8, 12, 16, 20$ respectively. Each collection spans exactly two organizations with $n_{required} = 2$, so the endorsing peer blocks until both members acknowledge receipt during proactive dissemination. The workload uses \texttt{PDC\_BENCH} with each transaction writing to a single randomly selected bilateral collection; all other parameters (block size, endorsement policy, value size) remain fixed. We use the same geo-distributed testbed as the dissemination experiment, adding peers to regions in round-robin order as the network grows.

This design exercises multiple scaling axes concurrently: the quadratic growth in collection definitions stresses the membership evaluation and endorsement policy checks during validation; the increasing number of peers amplifies block delivery fan-out and gossip protocol overhead; and the growing private state footprint (one hashed-state namespace per collection) tests state DB scalability. Because each bilateral collection has only two members and $n_{required} = 2$, per-collection dissemination cost remains constant---isolating the system-level overhead of managing a large collection topology from the per-collection dissemination cost already characterized in EQ3.
