\section{Evaluation}

We evaluate four questions: (1)~what is the end-to-end overhead of private data relative to all-public transactions? (2)~how sensitive is performance to dissemination parameters? (3)~how far does the system scale with collection count and network size? and (4)~how resilient is the system to membership changes and faults?

\noindent\textbf{Experimental Setup.}
We design four application workloads modeling multi-party business processes to stress different dimensions of the PDC mechanism (Table~\ref{table:workloads}). Supply Chain maximizes the collection count per organization; Auction isolates dissemination throughput; Trade Finance stresses scalability under strict majority endorsement; and EHR tests read throughput. For each, we compare a \textit{PDC variant} against an \textit{all-public variant} to isolate privacy overhead. In addition, we develop \texttt{PDC\_BENCH}, a minimal chaincode for controlled micro-benchmarking that exercises all public and private data APIs, enabling direct measurement of individual PDC operations against a public-ledger baseline.

\subsection{End-to-End Performance}
We compare the PDC and all-public variants of all four workloads at peak throughput, with roughly 40--50\% of transactions touching private data collections in each PDC variant. Figure~\ref{fig:e2e-throughput} shows throughput and commit latency side by side.
Across all four workloads, PDCs consistently reduce peak throughput: 3\% for Auction, 7.5\% for Supply Chain, 10\% for Trade Finance, and 14\% for EHR. The overhead correlates with per-transaction complexity.
EHR sees the largest drop because it is the most read-heavy workload (3 reads per tx) and each private-data read adds a private-data-store lookup. Trade Finance suffers from its strict 7-of-12 endorsement policy: the additional dissemination and commit-path work compounds an already long critical path. Auction and Supply Chain, with simpler profiles and lighter endorsement policies, see the smallest overheads.
Commit latency increases by 4--17\% for the first three workloads but jumps 40\% for Trade Finance, where the committer must verify private-data hashes and potentially trigger reactive retrieval on top of an already slow endorsement round.

\textit{Where does the overhead originate?} Per-node Prometheus metrics and disk I/O counters collected during each 300\,s run point to three sources.
(1)~\textit{CPU:} per-transaction cost rises 10--20\% due to hash computation, transient-store writes, and private-data-store flushes---none of which exist in the all-public path.
(2)~\textit{Network:} because the ordering service never sees private payloads, they must be distributed point-to-point via gossip, increasing per-transaction traffic by 10--15\%; missed payloads trigger reactive retrieval that further inflates peer-to-peer traffic.
(3)~\textit{Disk reads:} committers query the transient store and private-data store to match payloads against on-chain hashes, increasing read I/O by 2--4$\times$.
Notably, total disk \textit{writes} decrease: in the all-public path every peer writes the full payload, whereas with PDCs the ledger records only a hash and only authorized peers persist the private payload---a storage benefit of selective replication.

\subsection{Scalability}
To stress-test the per-collection overhead, we deploy a 10-organization WAN network and scale the number of private data collections attached to a single chaincode from 10 to 300 in increments of 50 (configurations c10--c300). The workload uses \texttt{PDC\_BENCH}: each transaction writes to a single randomly selected collection; all other parameters (block size, endorsement policy, value size) remain fixed. Figure~\ref{fig:scalability} plots throughput (bars) and per-block validation cost (line) as the collection count grows.
The system sustains peak throughput (740--760\,tx/s) up to 250 collections. Starting at c150, average transaction latency begins to stretch; at c300, throughput drops by 11\% (to 660\,tx/s) while average latency spikes to 24\,s. Disk I/O, ledger commit time, and state DB metrics remain flat throughout the range, confirming that the storage layer is \emph{not} the bottleneck. Instead, the dominant cost is CPU time spent in the validation phase: per-block validation cost rises linearly from 0.20\,s at c10 to 0.59\,s at c300 (Figure~\ref{fig:scalability}).

This degradation is \emph{not} an architectural limit but an implementation artifact. The root cause is a serialization shortcut: all collection definitions are stored in a single \texttt{CollectionConfigPackage} protobuf per chaincode, and the validator performs a full \texttt{proto.Unmarshal} of this package for every transaction that touches any collection---even though each transaction touches only one. At 300 collections, this deserialization runs hundreds of times per second, exhausting CPU headroom. Caching the parsed package or indexing individual collection policies would eliminate the linear penalty without architectural changes. Real-world deployments today rarely exceed a few dozen collections per chaincode, well within the flat-throughput region.

\subsection{Dissemination Sensitivity}
To isolate the effect of the dissemination parameters introduced in Algorithm~\ref{alg:dissemination}, we deploy a 20-organization WAN network (Table~\ref{table:workloads}) where all peers share a single private data collection.
The workload consists entirely of private-data write transactions, each requiring a single endorsement, running at 600\,tx/s for 300\,s.
We study two dissemination strategy families by varying the collection configuration parameters $n_{dissem}$ and $n_{required}$:
\begin{itemize}[topsep=0pt,noitemsep,leftmargin=15pt]
\item \textit{Pull-dominant:} The endorsing peer proactively pushes to $n_{dissem} \in \{0, 4, 9, 14, 19\}$ peers before ordering; the remaining $20 - n_{dissem}$ peers retrieve private data reactively at commit time ($n_{required} = 0$, so endorsement latency is unaffected by dissemination).
\item \textit{Push-with-quorum:} The endorsing peer pushes to all 20 peers ($n_{dissem} = 20$) but blocks the endorsement response until $n_{required} \in \{0, 4, 9, 14, 19\}$ peers acknowledge receipt.
\end{itemize}

Figure~\ref{fig:dissemination} shows performance under the two strategies. Under pull-dominant dissemination, throughput degrades as $n_{dissem}$ decreases (Figure~\ref{fig:dissemination}(a)): with $n_{dissem}=0$, only the endorsing peer holds the private data at commit time, so all 19 remaining peers issue pull requests. Because pulls target random peers, most miss the single data holder and time out. Transactions still commit---peers record the missing data for background reconciliation---but the low cumulative disk writes in Figure~\ref{fig:dissemination}(b) confirm that most peers have not yet persisted the private data within the measurement window.

Under push-with-quorum, all private data reaches every peer (disk writes remain uniformly high in Figure~\ref{fig:dissemination}(d)), but requiring WAN acknowledgments before returning the endorsement response creates backpressure. Figure~\ref{fig:dissemination}(c) shows throughput dropping 81\% at $n_{required}=19$ as the endorser waits for the slowest cross-region peer. Proposal latency rises correspondingly (Figure~\ref{fig:dissemination}(d)), confirming that the bottleneck shifts from commit-time retrieval to endorsement-time blocking.
We conclude that asynchronous push ($n_{dissem}=N{-}1, n_{required}=0$) is the optimal WAN configuration: it ensures all peers receive private data before commit time (full disk writes across all peers), maximizes throughput, and eliminates commit-time pull storms and deferred reconciliation.

\subsection{Dynamic Membership and Fault Tolerance}
We design a two-phase experiment on a five-organization Fabric network (one peer per org). In Phase~1, four organizations share a private data collection and populate it with 50,000 key-value pairs. In Phase~2, the collection policy is expanded to a fifth organization via a chaincode upgrade; the new peer comes online and reconciles the 50,000 historical records it is now entitled to while an additional 50,000 transactions execute concurrently.

We run this scenario under two conditions: (i)~a \textit{baseline} with no faults, and (ii)~a \textit{stress} configuration that injects compound failures during reconciliation---killing the newly joined peer mid-reconciliation and crashing an existing data-holder peer---each followed by recovery after a delay. Correctness is verified by fetching all 100,000 key-value pairs from every peer after stabilization.

\textit{Baseline Performance Isolation.}
As shown in Figure~\ref{fig:dynamic-membership}(a), throughput rises from 396\,tx/s to 474\,tx/s when the fifth organization joins, reflecting the added endorsement capacity. The reconciliation curve shows that the background synchronization of 50,000 historical records progresses smoothly alongside active transaction processing, completing at $t \approx 229$\,s without visible throughput degradation. This confirms that reconciliation is effectively isolated from the transaction execution pipeline.

\textit{Resilience Under Compound Failures.}
Figure~\ref{fig:dynamic-membership}(b) shows behavior under adversarial conditions. When the synchronizing peer is crashed (\texttt{-org5}), reconciliation halts immediately (plateau in the green curve) but resumes from its last synchronized block upon recovery (\texttt{+org5}).

When an existing data-holder peer is crashed (\texttt{-org2}), throughput drops from the 5-peer peak of ${\sim}500$\,tx/s to the 4-peer baseline of ${\sim}400$\,tx/s as the client load redistributes among remaining endorsers. Despite the newly joined peer being offline for 30 seconds and the loss of multiple endorsement sources, reconciliation reaches 100\% data consistency at $t \approx 259$\,s. The gossip-based pull mechanism is self-healing and robust to compound failures during synchronization.
