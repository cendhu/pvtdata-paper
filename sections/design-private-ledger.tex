\section{Design of Private Data Transactions}
\label{s:design}
The simulate-order-validate-commit protocol, as argued in the previous section, provides the right foundation for private data. We now describe how private data transaction processing is built on top of it. The design centers on two principles: a \textit{dual-layered storage strategy} that separates raw private data from its cryptographic footprint, and \textit{partitioned state management} that isolates private data while preserving global verifiability. We assume a key-value data model for both public and private state, where each state entry is a $(key, value)$ pair identified by a unique key within a namespace, and each key carries a version that is updated whenever the key is written by a committed transaction. Figure~\ref{fig:pdc-flow} provides an end-to-end overview of the private data collection transaction flow, with numbered steps spanning all four phases; Peer~1 and Peer~2 are members of collection $\mathcal{C}_1$, while Peer~n is not. The following subsections detail each component.

\subsection{Partitioned State Management}
Integrating private data into a shared ledger without exposing it to unauthorized peers during validation is the central challenge. The solution is to partition the state database into three logical domains. Each smart contract is allocated a dedicated \textit{namespace} within these domains; access and modification within a namespace are governed by an endorsement policy specifying the set of authorized peers whose signatures are required to validate state changes. The three partitions for each namespace are:
\begin{itemize}[topsep=0pt,noitemsep,leftmargin=15pt]
    \item \textit{Public State:} Accessible to all nodes; contains the standard replicated ledger state.
    \item \textit{Hashed State:} Replicated across all nodes; contains cryptographic hashes of private keys and values, providing the evidence required for tamper-detection and serializability checks.
    \item \textit{Private State:} Isolated to authorized peers; contains the actual private key-value pairs. Modifications are governed by the collection's own endorsement policy $\mathcal{E}$, which supersedes the smart contract's default endorsement policy for private data operations.
\end{itemize}
Public State and Hashed State are consistent across all peers in the network, providing a globally uniform view of the ledger. Private State, by contrast, varies from peer to peer depending on which collections each peer's organization belongs to---a peer only maintains private state for collections in which it is authorized. By confining all private data interactions to smart contract namespaces, this partitioning satisfies \textbf{R3} (Smart Contract Controlled Access); the replicated Hashed State that mirrors Private State reinforces \textbf{R2} (Hash-Data Integrity).

\subsection{Transaction Simulation}
\input{sections/apis.tex}
The simulation phase (steps 1--4 in Figure~\ref{fig:pdc-flow}) begins when a client submits a transaction proposal to a subset of peers that are members of the relevant collections. Restricting execution to authorized peers directly satisfies \textbf{R4} (Authorized Execution Only). Private data enters the system through a \textit{transient data} field---present during execution but excluded from the final transaction payload, so it is never broadcast to all peers during ordering.

Upon receiving the proposal, each endorsing peer executes the smart contract using the APIs defined in Table~\ref{table:apis}. Public data operations (\texttt{GetState}, \texttt{PutState}, \texttt{DelState}) access the standard replicated state; private data operations (\texttt{GetPrivateData}, \texttt{PutPrivateData}, \texttt{DelPrivateData}) access collection-specific private state. When a transaction reads and writes both public and private state, the endorsing peers must satisfy both the smart contract's endorsement policy (for public writes) and the collection's policy $\mathcal{E}$ (for private writes). Private writes also perform \textit{dual-recording}: \texttt{PutPrivateData}$(c, k, v)$ writes $(k, v)$ to the private write-set and simultaneously records $(H(k), H(v))$ in the hashed write-set, satisfying \textbf{R2} (Hash-Data Integrity). Because each endorsing peer independently computes the hash from the private data it receives, the collection's endorsement policy $\mathcal{E}$ bounds the trust assumption: as long as at least one honest endorser participates, the recorded hash faithfully reflects the private data. If all authorized members collude to endorse a false hash, they can corrupt the audit trail for that collection, but cannot violate public state invariants or affect collections to which they do not belong; a full analysis appears in \S\ref{s:threat-model}.

On successful execution, the peer produces three partitioned read-write sets: a \textit{public read-write set} for replicated ledger state, a \textit{hashed read-write set} containing cryptographic hashes of private keys and values, and a \textit{private write-set} per collection involved in the transaction. Table~\ref{table:apis} illustrates concretely how a sequence of API calls generates these outputs. The endorsing peer signs the public and hashed read-write sets and returns them to the client as part of its endorsement response; the private write-sets, meanwhile, are stored in the peer's transient store and proactively disseminated to other collection members as described in \S\ref{s:gossip}.

Since transaction order is determined only after simulation completes, committing private data to permanent storage at this point would risk corrupting committed state with transactions that are later invalidated. Peers therefore hold private write-sets in a \textit{transient store} until the transaction commits, satisfying \textbf{R1} (Transactional Atomicity and Isolation). Entries are keyed by transaction identifier ($txid$) to prevent collisions among concurrently simulated transactions. The transient store is backed by a durable on-disk store (LevelDB); if a peer crashes and restarts, any write-sets that survived to disk remain available. For write-sets lost due to incomplete flushes, the peer treats them as missing at commit time and recovers them via the reactive retrieval protocol (Algorithm~\ref{alg:retrieval}), so transient store durability is a performance optimization---not a correctness requirement.

When a smart contract reads private data that is absent from the local store (e.g., due to incomplete dissemination), \texttt{GetPrivateData} returns an error and the simulation fails immediately. Only fully informed simulations yield valid endorsements.

\subsection{Transaction Validation and Commit}
Once the client collects sufficient endorsements, it submits the transaction---containing the public and hashed read-write sets, but not the private write-sets---to the ordering service, which batches transactions into blocks and delivers them to all peers (steps 5--7 in Figure~\ref{fig:pdc-flow}). The ordering phase is unchanged from the base protocol; the ordering service never sees private data.

The validation (step 8) and commit (steps 9--10) phases accept only correctly endorsed, serializable transactions. Endorsement policy checks across transactions in a block can run in parallel; OCC validation proceeds sequentially in committed order, since validating one transaction affects the state versions visible to subsequent ones. A serialization graph can alternatively enable parallel validation of non-conflicting transactions~\cite{perf-concurrency}.

Each transaction's endorsements are checked against both the smart contract's policy (public writes) and the collection's policy $\mathcal{E}$ (private writes), satisfying \textbf{R4} (Authorized Execution Only). OCC checks then compare the transaction's read versions against the current Public State and Hashed State. Every write to a private key $k$ atomically updates its hash $H(k)$ in the replicated ledger, so Hashed State version identifiers are sufficient to detect all conflicts on private data without exposing the data itself. \textit{Blind validation} follows directly: every peer can verify serializability of private state changes without ever seeing the private data. Table~\ref{table:apis} makes this concrete---a stale read on $k_2$ in collection $c_1$ is caught through hashed state version comparison alone, even by a validator with no access to the underlying private values.

We now formalize this correctness claim. We adopt the standard OCC serializability criterion~\cite{occ}: a transaction $T_i$ is \textit{serializable} with respect to a committed history if no key in $T_i$'s read-set was modified by any transaction $T_j$ that committed after $T_i$ began---equivalently, the read-set versions recorded at simulation time must match the current committed versions at validation time. Let $\mathcal{S}_P$ denote the Private State, $\mathcal{S}_H$ the Hashed State, and $\text{ver}(s, k)$ the version of key $k$ in state $s$.

\noindent\textbf{Theorem 1} (Conflict-Complete Proxy). \textit{Assuming $H$ is a collision-resistant hash function (so that $H(k_1) = H(k_2) \implies k_1 = k_2$), the Hashed State $\mathcal{S}_H$ is a conflict-complete proxy for the Private State $\mathcal{S}_P$: for any transaction $T_i$ involving private data, an OCC conflict exists in $\mathcal{S}_P$ if and only if a corresponding conflict exists in $\mathcal{S}_H$. Formally, for every private key $k$ in collection $c$:}
$$\text{ver}(\mathcal{S}_P, k) \neq \text{ver}_{\text{sim}}(T_i, k) \iff \text{ver}(\mathcal{S}_H, H(k)) \neq \text{ver}_{\text{sim}}(T_i, H(k))$$

\textit{Proof.} We establish three invariants maintained by the API (Table~\ref{table:apis}):
\begin{enumerate}[topsep=0pt,noitemsep,leftmargin=15pt]
    \item \textit{Write Bijection:} By construction, every \texttt{PutPrivateData}$(c, k, v)$ atomically produces both a private write-set entry $(k, v)$ and a hashed write-set entry $(H(k), H(v))$. The commit procedure assigns both entries the same version identifier (block number, transaction index). \texttt{DelPrivateData} and \texttt{PurgePrivateData} similarly produce corresponding hashed entries. Thus, $\text{ver}(\mathcal{S}_P, k)$ and $\text{ver}(\mathcal{S}_H, H(k))$ are updated in lockstep: $\text{ver}(\mathcal{S}_P, k) = \text{ver}(\mathcal{S}_H, H(k))$ after every committed transaction.
    \item \textit{Read Recording:} Every \texttt{GetPrivateData}$(c, k)$ records $\text{ver}(\mathcal{S}_H, H(k))$ in the hashed read-set. Since Invariant~1 guarantees $\text{ver}(\mathcal{S}_P, k) = \text{ver}(\mathcal{S}_H, H(k))$, this recorded version is identical to the version that would appear in a hypothetical private read-set.
    \item \textit{Conflict Detection:} During OCC validation, a conflict on private key $k$ exists iff $\text{ver}(\mathcal{S}_P, k)$ has changed since simulation. By Invariant~1, $\text{ver}(\mathcal{S}_P, k)$ changes iff $\text{ver}(\mathcal{S}_H, H(k))$ changes. By Invariant~2, the validator compares $\text{ver}(\mathcal{S}_H, H(k))$ against the same baseline. Therefore, the OCC check on $\mathcal{S}_H$ detects a conflict iff the corresponding check on $\mathcal{S}_P$ would.
\end{enumerate}
Since both directions hold, blind validation produces neither false negatives (missed conflicts) nor false positives (spurious aborts). \hfill $\square$

\noindent\textit{Edge Case: Concurrent Membership Changes.}
A collection's membership policy $\mathcal{M}$ may be updated in the same block that contains private data transactions for that collection. Whether processed sequentially or via a serialization graph~\cite{perf-concurrency}, a membership change at transaction index $j$ takes effect for all subsequent transactions $j' > j$ within the same block. This within-block ordering is a deliberate design choice: it avoids the complexity of batch-level policy resolution and guarantees deterministic validation across all peers. Endorsement and membership checks during validation therefore always reflect the correct policy version. A newly added member will not yet have historical private data at validation time; the reactive retrieval and reconciliation mechanisms (\S\ref{s:gossip}) handle this asynchronously without affecting current-block validity.

Theorem~1 and the sequential policy application ensure \textbf{R1} across public and private state.

Once a transaction is successfully validated, the private write-sets are moved from the transient store to the permanent private data store. The Private State within a smart contract's namespace reflects only the latest value for each key, overwritten by each subsequent committed transaction. To support historical retrieval, a separate \textit{private data store} retains all historical private write-sets indexed by block number and transaction index, so that when a new member joins a collection, the complete history of private data can be synchronized to that member.

\noindent\textit{Handling Missing Private Data.}
A peer may not possess all private write-sets at commit time. During commit, each peer classifies missing private data into two categories: \textit{eligible missing data}---private data for collections in which the peer's organization is a member but the data has not yet arrived (e.g., due to network delays during dissemination)---and \textit{ineligible missing data}---private data for collections in which the peer's organization is not currently a member. In both cases the peer records the block number, transaction index, collection identifier, and the corresponding hashed key-value pairs from the hashed write-set; the stored hashes allow the peer to verify the authenticity of private data received later during retrieval. Eligible missing entries are fetched via the reactive retrieval protocol described in \S\ref{s:gossip}, satisfying \textbf{R7} (Data Availability). Ineligible entries are retained because if the organization later joins the collection, it can use them to identify and synchronize historical private data it is now authorized to access, supporting \textbf{R5} (Dynamic Membership).

\noindent\textit{Data Lifecycle Management.}
The private data store supports data lifecycle management through the collection's block-to-live (BTL) retention parameter $\tau$, satisfying \textbf{R6} (Data Lifecycle Management). Any private data committed more than $\tau$ blocks ago is considered expired. To comply with privacy regulations such as the ``Right to Forget,'' a background process periodically purges expired private data while preserving the on-chain hashes, keeping the historical audit trail tamper-evident. The implications of retaining these on-chain hashes are discussed in \S\ref{s:threat-model}.

The design above assumes that private write-sets reach authorized peers before commit time. Ensuring this in practice---and recovering data when it does not arrive in time---is the subject of the next section, which addresses \textbf{R5} and \textbf{R7}.
