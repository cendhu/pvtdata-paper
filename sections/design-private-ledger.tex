\textbf{Transient Data in Transaction Proposal.}
Each transaction in the blockchain contains the client's input to the smart contract computation, for auditing.
However, client input to smart contract computation might be confidential and thus shouldn't appear in the blockchain's blocks.
\\Otherwise, non-authorized observers could then easily obtain sensitive information such as encryption keys, raw biometric data, social security numbers, and more.

Encrypting this secret information at the client side and decrypting it in the smart contract, is problematic:
\begin{itemize}
	\item Using symmetric encryption induces either chicken-and-egg problem or the need to send these keys separately, complicating the interaction between clients and peers.
	\item It is tempting to use asymmetric encryption and have the client encrypt the secrets using the public key of the peers, unfortunately since semantically secure asymmetric encryption has to be non deterministic \cite{probEncryption} this limits the use only for endorsement policies that require signatures from peers with the same private keys, making it a non viable candidate. 
\end{itemize}

Instead, the client passes the secret information to the peer in a different data section, denoted as \emph{transient data} than the public input to the smart contract computation which is omitted from the transaction that ends up in the blockchain.

In order to preserve some minimal level of auditing capabilities, the peer includes the hash of all data sent from the client (including the transient data) in the input to the signature over the smart contract computation results.

This allows a client to prove to an auditor that the computation was either done correctly or incorrectly by knowing the transient data.

\textbf{Hashed, and Private Read Write Set.}
\\ Owner: Manish

\par
While simulating a transaction, in addition to the read-write set for the public data, 
we generate another read-write set for the hashes of the private data referred to as \textit{hashed read-write set}.
Further, we produce a separate write-set for each private data collection that the transaction modifies. We refer to this write-set as \textit{private write-set}.
A private write-set contains only writes for the private data for a specific collection.

\par
During simulation of a transaction, for each key in a private data collection that the transaction reads,
we add a tuple <collection name, hash of the key, version of the hash of the key> to the hashed read-set.
Further, for each key in a private data collection that the transaction creates or modifies, 
we add a tuple <collection name, hash of the key, hash of the new value>
to the hashed write-set and in addition, we add a tuple <pre-images of the private key, pre-images of the new value>
to the collection specific private write-set.

Thus, we generate one public read-write set, one hashed read-write set and one or more private write-sets for a simulation of a transaction.

\par
Like the public read-write set, we package the hashed read-write set with the transaction data inside the block. However, we share
each private write-sets associated with the transaction with only blockchain nodes that belong to the organizations
that are member of the corresponding private data collection.
