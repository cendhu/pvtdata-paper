\textbf{Transient Data in Transaction Proposal.}
Each transaction in the blockchain contains the client's input to the smart contract computation, for auditing.
However, client input to smart contract computation might be confidential and thus shouldn't appear in the blockchain's blocks.
\\Otherwise, non-authorized observers could then easily obtain sensitive information such as encryption keys, raw biometric data, social security numbers, and more.

Encrypting this secret information at the client side and decrypting it in the smart contract, is problematic:
\begin{itemize}
	\item Using symmetric encryption induces either chicken-and-egg problem or the need to send these keys separately, complicating the interaction between clients and peers.
	\item It is tempting to use asymmetric encryption and have the client encrypt the secrets using the public key of the peers. This requires the client to encrypt the data separately for each peer that is involved in the smart contract execution, however requires infrastructure for public key discovery.
\end{itemize}

Instead, the client passes the secret information to the peer in a different data section, denoted as \emph{transient data} than the public input to the smart contract computation which is omitted from the transaction that ends up in the blockchain.

In order to preserve some minimal level of auditing capabilities, the peer includes the hash of all data sent from the client (including the transient data) in the input to the signature over the smart contract computation results.

This allows a client to prove to an auditor that the computation was either done correctly or incorrectly by knowing the transient data.

\textbf{Hashed, and Private Read Write Set.}
\\ Owner: Manish

\par
While simulating a transaction, in addition to the read-write set for the public data, 
we generate another read-write set for the hashes of the private data referred to as \textit{hashed read-write set}.
Further, we produce a separate write-set for each private data collection that the transaction modifies. We refer to this write-set as \textit{private write-set}.
A private write-set contains only writes for the private data for a specific collection.

\par
During simulation of a transaction, for each key in a private data collection that the transaction reads,
we add a tuple <collection name, hash of the key, version of the hash of the key> to the hashed read-set.
Further, for each key in a private data collection that the transaction creates or modifies, 
we add a tuple <collection name, hash of the key, hash of the new value>
to the hashed write-set and in addition, we add a tuple <pre-images of the private key, pre-images of the new value>
to the collection specific private write-set.

Thus, we generate one public read-write set, one hashed read-write set and one or more private write-sets for a simulation of a transaction.

\par
Like the public read-write set, we package the hashed read-write set with the transaction data inside the block. However, we share
each private write-sets associated with the transaction with only blockchain nodes that belong to the organizations
that are member of the corresponding private data collection.

\textbf{Serializability Validation.}
\\ Owner: Manish
\par
In order to maintain serializability of the transactions, we validate each transaction in the block,
\textit{logically} in the order in which they apper in the block. In order to perform validation for a transaction,
we  verify that the version of the each of key hashes in the hashed read-set are the same as in the committed state
as of the immidiate preceding transaction. If the transaction is found to be valid, then we apply to the committed state 
the write-set portion of the hashed read-write-set and then we proceed with the validation of the next transaction in the block.
\par
This approach allows for the peers across organizations to perform validation of the transactions in a deterministic manner
without having to access to the actual private data. As a result, this approach allows for modifying in a single transaction,
the public data and the private data from more than one collections. However, this approach comes with a limitation, which is,
unlike public data, we can only support the point queries on the private data with a serializability guarantee.
The support of complex queries would work only where applications can tolerate with repeatable read semantics 
for the transaction isolation.

\textbf{Private Data Store.}
\\ Owner: Manish
\par
We prserve each of the private write-set associated with a collection within a transaction in a separate store
that we refer to as \textit{ private data store}. Since each organization may be eligible to receive the private 
write-sets for a different set of collections in a transaction, unlike block store, the data stored in the 
private data store on one peer may differ from the data stored into another peer with in a network. The primary
motivation for the private data store is to maintain all the versions of private data for the provenance related queries.
The private data store also serves the request from other organizations for the private write-sets associated 
with a transaction. However, the private data store serves the write-sets of only those collections in a transaction for which
the requesting organization is a member.

\par
As noted before, at the time of committing a block on a peer, some of the private write-sets may be not be available 
corresponding to the transactions included in the block. There are two primary reasons for non availability of the 
private write-sets for a collection at the time of committing a block; 1) the organization is not eligible for one or
more collections modified in a transaction and 2) the committing peer could not successfully access to the private
write-sets. In the private data store, in addition to persisting the private write-sets, we also maintain this information
about the non-availability of the private write-sets along with a notion that specifies one of the above two mentioned reasons.
This information is used for attempting to retrieve the missing private write-sets in the future under two scenarios; 
1) the peer periodically tries to fetch the private write-sets for which the peer is eligible but the peer could not retrieve 
this data from other peers at the time of commit and 2) if the peer becomes eligible for a collection in the future. We discuss
these scenarios in more detail in the latter sub-sections.
