\section{Design of Private Data Transactions}
\label{s:design}
The simulate-order-validate-commit protocol, as argued in the previous section, provides the right foundation for private data. We now describe how private data transaction processing is integrated within it.
The design centers on three ideas: \textit{partitioned state management} that splits state into public, hashed, and private domains; \textit{blind validation} that uses the hashed partition as a conflict-complete proxy, enabling serializability checks without accessing private data; and \textit{decoupled dissemination} that separates private data distribution from block ordering, with proactive push, reactive pull, and background reconciliation ensuring eventual availability. We assume a key-value data model for both public and private state, where each state entry is a $(key, value)$ pair identified by a unique key within a namespace,
and each key carries a version that is updated whenever the key is written by a committed transaction. Figure~\ref{fig:pdc-flow} provides an end-to-end overview of the private data collection transaction flow,
with numbered steps spanning all four phases; Peer~1 and Peer~2 are members of collection $\mathcal{C}_1$, while Peer~n is not. The following subsections detail each component.

\subsection{Partitioned State Management}
Integrating private data into a shared ledger without exposing it to unauthorized peers during validation is the central challenge. The solution is to partition the state database into three logical domains. Each smart contract is allocated a dedicated \textit{namespace} within these domains; access and modification within a namespace are governed by an endorsement policy (\S\ref{ss:arch-req}) specifying the set of authorized peers whose signatures are required to validate state changes. The three partitions for each namespace are:
\begin{itemize}[topsep=0pt,noitemsep,leftmargin=15pt]
    \item \textit{Public State:} Accessible to all nodes; contains the standard replicated ledger state.
    \item \textit{Hashed State:} Replicated across all nodes; contains cryptographic hashes of private keys and values, providing the evidence required for tamper-detection and serializability checks.
    \item \textit{Private State:} Isolated to authorized peers; contains the actual private key-value pairs, which are the pre-images of the hashed state. Modifications are governed by
      the collection's own endorsement policy $\mathcal{E}$, which supersedes the smart contract's default endorsement policy. In practice, $\mathcal{E}$ is defined over a subset of (or all of) the collection's membership $\mathcal{M}$, since only members possessing the private data can meaningfully endorse transactions.
\end{itemize}
Public State and Hashed State are consistent across all peers in the network, providing a globally uniform view of the ledger. Private State, by contrast, varies from peer to peer depending on which collections each peer's organization belongs to---a peer only maintains private state for collections in which it is authorized. By confining all private data interactions to smart contract namespaces, this partitioning satisfies \textbf{R3} (Smart Contract Controlled Access); the replicated Hashed State that mirrors Private State reinforces \textbf{R2} (Hash-Data Integrity).

\subsection{Transaction Simulation}
The simulation phase (steps 1--4 in Figure~\ref{fig:pdc-flow}) begins when a client submits a transaction proposal to multiple endorsing peers to collect adequate endorsements as per the endorsement policies defined for the smart contract and the private data collections involved. Restricting execution to authorized peers directly satisfies \textbf{R4} (Authorized Execution Only). Private data enters the system through a \textit{transient data} field---present during execution but never becoming part of the replicated ledger.

The goal of the simulation phase is to collect read-write sets. Reads capture the data version read by the smart contract to make decisions on the proposed changes (i.e., writes). These read-write sets are subsequently used by optimistic concurrency control (OCC) based serializability checks~\cite{occ} in the validation phase. Since state is partitioned into public, hashed, and private domains (\S\ref{s:design}), the simulation produces three corresponding read-write sets---one per partition: a \textit{public read-write set} for replicated ledger state, a \textit{hashed read-write set} containing cryptographic hashes of private keys and values, and a \textit{private write-set} per collection involved in the transaction. Note that there is no private read-set; private reads are recorded in the hashed read-set via their hash versions to enable blind validation. A separate private read-set would force member and non-member peers to follow divergent validation paths---members checking private versions while non-members could not---breaking the uniform validation guarantee that lets every peer independently reach the same commit decision.

Upon receiving the proposal, each endorsing peer executes the smart contract using the APIs defined in Table~\ref{table:apis}. Public data operations (\texttt{GetState}, \texttt{PutState}, \texttt{DelState}) populate the public read-write set. Private data operations populate the hashed and private partitions: \texttt{GetPrivateData}$(c, k)$ records $(H(k), \text{version})$ in the hashed read-set; \texttt{PutPrivateData}$(c, k, v)$ performs \textit{dual-recording}, writing $(k, v)$ to the private write-set and simultaneously recording $(H(k), H(v))$ in the hashed write-set, satisfying \textbf{R2} (Hash-Data Integrity). \texttt{DelPrivateData} and \texttt{PurgePrivateData} similarly produce corresponding entries in both the private and hashed write-sets. Table~\ref{table:apis} illustrates the process end-to-end: transaction $T_i$ calls \texttt{GetState}($k_1$) followed by \texttt{PutState}($k_1$, $v_2$), producing a public read-set entry $(k_1, \text{v1})$ and a public write-set entry $(k_1, v_2)$. It then calls \texttt{GetPrivateData}($c_1$, $k_2$) and \texttt{GetPrivateData}($c_2$, $k_3$), which record $(H(k_2), \text{v5})$ and $(H(k_3), \text{v5})$ in the hashed read-set. Finally, \texttt{PutPrivateData}($c_1$, $k_2$, $v_3$) and \texttt{PutPrivateData}($c_2$, $k_3$, $v_4$) each perform dual-recording: $(k_2, v_3)$ and $(k_3, v_4)$ go to the private write-sets, while $(H(k_2), H(v_3))$ and $(H(k_3), H(v_4))$ go to the hashed write-set. When a transaction reads and writes both public and private state, the endorsing peers must satisfy both the smart contract's endorsement policy (for public writes) and the collection's policy $\mathcal{E}$ (for private writes).

\noindent\textbf{Post-Simulation: Transient Storage and Dissemination.}
Since transaction order is determined only after simulation completes, committing private data to permanent storage at this point would risk corrupting committed state with transactions that are later invalidated. Peers therefore hold private write-sets in a \textit{transient store} until the transaction commits, satisfying \textbf{R1} (Transactional Atomicity and Isolation).
As described in Algorithm~\ref{alg:dissemination}, after simulation the endorsing peer immediately pushes each private write-set to a subset of authorized collection members. The endorser does \textit{not} return its signed response to the client until at least $n_{required}$ peers have acknowledged receipt. Blocking on these acknowledgments guarantees that the data exists on multiple peers before the client can submit the transaction for ordering. After sending the endorsement response, a background gossip task continues pushing the write-set to the remaining peers up to $n_{dissem}$ total, spreading coverage without adding to client-observed latency.
We separate $n_{required}$ from $n_{dissem}$ so that operators can tune the two concerns independently. Setting $n_{dissem}$ aggressively ensures most collection members receive the data proactively, which minimizes commit-time reactive pulls; keeping $n_{required}$ small protects endorsement latency.
When the endorser cannot obtain $n_{required}$ acknowledgments before the timeout, it returns a failure to the client, who may retry with different endorsers. We deliberately do not retry dissemination at the endorser: keeping the endorsement critical path simple avoids unbounded retry delays and pushes fault handling to the client, where it belongs.

\subsection{Transaction Ordering}
Once the client collects sufficient endorsements, it submits the transaction---containing the public and hashed read-write sets, but not the private write-sets---to the ordering service (steps 5--7 in Figure~\ref{fig:pdc-flow}). The ordering service is a black box with respect to transaction content: it never sees private data. Its sole responsibility is to order transactions into blocks using consensus (CFT or BFT), and the block order determines the commit order. Its functionality is unchanged regardless of whether a transaction involves public data, private data, or both.

\subsection{Validation and Commit}
\noindent\textbf{Validation.}
Validation (step 8 in Figure~\ref{fig:pdc-flow}) proceeds in two sequential checks for each transaction in a block. First, each transaction's endorsement signatures are verified against the smart contract's endorsement policy for public writes and the collection's endorsement policy $\mathcal{E}$ for private writes. A transaction that involves both public and private state must satisfy both policies. Any transaction that fails this check is immediately marked invalid and skipped, satisfying \textbf{R4} (Authorized Execution Only). Endorsement policy checks across transactions in a block can run in parallel, since they depend only on the signatures attached to each transaction and not on mutable state.
Transactions that pass endorsement validation advance to OCC serializability checks~\cite{occ}, which compare the read-set versions recorded at simulation time against the current committed versions in the Public State and Hashed State. A transaction is serializable if no key in its read-set was modified by an intervening committed transaction. OCC validation proceeds sequentially in block order, since validating one transaction may update state versions visible to subsequent ones. A serialization graph can alternatively enable parallel validation of non-conflicting transactions~\cite{perf-concurrency, sparse-peer}.
Every write to a private key $k$ atomically updates its hash $H(k)$ in the replicated Hashed State, so hashed version identifiers are sufficient to detect all conflicts on private data without exposing the data itself. \textit{Blind validation} follows directly: every peer can verify serializability of private state changes without ever seeing the private data.

\noindent\textit{Walkthrough.}
Continuing the Table~\ref{table:apis} example: before $T_i$ reaches validation, another transaction $T_j$ commits and updates $k_2$ in $c_1$, advancing its hashed version from v5 to v8. At validation time, the validator compares $T_i$'s recorded read versions against current disk state: public key $k_1$ still has version~v1 (\cmark), hashed key $H(k_3)$ in $c_2$ still has version~v5 (\cmark), but hashed key $H(k_2)$ in $c_1$ now has version~v8, which does not match $T_i$'s recorded version~v5 (\xmark). The validator detects the stale read and marks $T_i$ as invalid---all without accessing any private value.
We formalize this correctness claim after describing the commit procedure below.

\noindent\textbf{Commit.}
For each valid transaction, the committer retrieves the corresponding private write-sets from the local transient store. If the data is unavailable locally, the committer triggers a reactive pull (Algorithm~\ref{alg:retrieval}): it first targets the transaction's endorsers, who are guaranteed to possess the write-sets, then falls back to the gossip network. Each received write-set is verified against the on-chain hash before acceptance, preventing forged data injection (\S\ref{s:threat-model}). Validated private data is moved to the permanent \textit{private data store}, which retains all historical write-sets indexed by block number and transaction index to support synchronization when new members join.
If reactive retrieval fails within the timeout, the peer commits the block anyway to preserve liveness, satisfying \textbf{R7} (Data Availability). Missing entries are recorded as \textit{eligible missing data}---block number, transaction index, collection identifier, and hashed key-value pairs---so that background reconciliation can retry (\S\ref{s:dynamic-membership}); the stored hashes allow later verification. Until reconciliation completes, \texttt{GetPrivateData} for a missing key returns an error. Private data for collections in which the peer is \textit{not} a member is tracked as \textit{ineligible missing data}; these are never fetched but retained so that if the organization later joins, they can be reclassified to trigger synchronization, supporting \textbf{R5} (Dynamic Membership). The reverse transition is handled during membership updates (\S\ref{s:dynamic-membership}).

\noindent\textbf{Proof of Validation Correctness.}
We now formalize the correctness of blind validation. We adopt the standard OCC serializability criterion~\cite{occ}: a transaction $T_i$ is \textit{serializable} with respect to a committed history if no key in $T_i$'s read-set was modified by any transaction $T_j$ that committed after $T_i$ began---equivalently, the read-set versions recorded at simulation time must match the current committed versions at validation time. Let $\mathcal{S}_P$ denote the Private State, $\mathcal{S}_H$ the Hashed State, and $\text{ver}(s, k)$ the version of key $k$ in state $s$. We call $\mathcal{S}_H$ a \textit{conflict-complete proxy} for $\mathcal{S}_P$ if every OCC conflict in $\mathcal{S}_P$ is detectable in $\mathcal{S}_H$ and vice versa---that is, the two states agree on which transactions conflict, so validating against $\mathcal{S}_H$ is equivalent to validating against $\mathcal{S}_P$.

\noindent\textbf{Theorem 1} (Conflict-Complete Proxy). \textit{Assuming $H$ is a collision-resistant hash function (so that $H(k_1) = H(k_2) \implies k_1 = k_2$), the Hashed State $\mathcal{S}_H$ is a conflict-complete
proxy for the Private State $\mathcal{S}_P$: for any transaction $T_i$ involving private data, an OCC conflict exists in $\mathcal{S}_P$ if and only if a corresponding conflict exists in $\mathcal{S}_H$.
Formally, for every private key $k$ in collection $c$:} \\
$\text{ver}(\mathcal{S}_P, k) \neq \text{ver}_{\text{sim}}(T_i, k) \iff \text{ver}(\mathcal{S}_H, H(k)) \neq \text{ver}_{\text{sim}}(T_i, H(k))$
\\
\textit{Proof.} We first establish a key invariant. By construction, every \texttt{PutPrivateData}$(c, k, v)$ atomically produces both a private write-set entry $(k, v)$ and a hashed write-set entry $(H(k), H(v))$. The commit procedure assigns both entries the same version identifier (block number, transaction index). \texttt{DelPrivateData} and \texttt{PurgePrivateData} similarly produce corresponding hashed entries. Thus, $\text{ver}(\mathcal{S}_P, k)$ and $\text{ver}(\mathcal{S}_H, H(k))$ are updated in lockstep: $\text{ver}(\mathcal{S}_P, k) = \text{ver}(\mathcal{S}_H, H(k))$ after every committed transaction (\textit{Write Bijection}). Furthermore, every \texttt{GetPrivateData}$(c, k)$ records $\text{ver}(\mathcal{S}_H, H(k))$ in the hashed read-set, which by Write Bijection equals $\text{ver}(\mathcal{S}_P, k)$ at simulation time. We now prove each direction.

($\Rightarrow$) Suppose a conflict exists in $\mathcal{S}_P$: some transaction $T_j$ committed after $T_i$'s simulation and changed $\text{ver}(\mathcal{S}_P, k)$. By Write Bijection, $T_j$'s commit also advanced $\text{ver}(\mathcal{S}_H, H(k))$ by the same amount. Since the hashed read-set recorded the pre-$T_j$ version, the OCC check on $\mathcal{S}_H$ detects the mismatch.

($\Leftarrow$) Suppose a conflict exists in $\mathcal{S}_H$: $\text{ver}(\mathcal{S}_H, H(k))$ differs from the version recorded in $T_i$'s hashed read-set. By Write Bijection, $\text{ver}(\mathcal{S}_H, H(k))$ can only change when a transaction commits a write to private key $k$, which simultaneously advances $\text{ver}(\mathcal{S}_P, k)$. Therefore, $\text{ver}(\mathcal{S}_P, k)$ has also changed since simulation, confirming a conflict in $\mathcal{S}_P$.

\noindent Since both directions hold, blind validation produces neither false negatives (missed conflicts) nor false positives (spurious aborts).

\subsection{Membership and Data Lifecycle}
\label{s:dynamic-membership}

\noindent\textbf{Reconciliation and Dynamic Membership.}
The background reconciliation service periodically retries fetching eligible missing data recorded during commit, using the same hash-based verification as Algorithm~\ref{alg:retrieval}.
When a new member is added to a collection's policy $\mathcal{M}$, the peer detects the change during block commit and reclassifies all \textit{ineligible missing data} for that collection as \textit{eligible}, triggering reconciliation to synchronize historical write-sets.
Removal is symmetric: the departing peer reclassifies its \textit{eligible missing data} as \textit{ineligible}, stopping reconciliation. It retains private data committed before removal---preserving auditability---but the updated policy excludes it from future dissemination and retrieval. The departing peer can still serve pull requests from current members for historical data it holds, since on-chain hashes let the requester verify authenticity regardless of the source's current membership. If the organization is later re-added, ineligible entries revert to eligible, triggering reconciliation as in the join case.

\noindent\textbf{Data Lifecycle Management.}
\label{s:lifecycle}
The private data store supports two complementary mechanisms for data lifecycle management, satisfying \textbf{R6} (Data Lifecycle Management).

\textit{Automatic expiry.} Each collection defines a block-to-live (BTL) retention parameter $\tau$. Any private data committed more than $\tau$ blocks ago is considered expired. A background process periodically purges expired private data while preserving the on-chain hashes, keeping the historical audit trail tamper-evident.

\textit{Explicit purge.} For finer-grained control---e.g., to comply with the Right to be Forgotten under the EU GDPR~\cite{gdpr} on a per-key basis---the \texttt{Purge\-Private\-Data}$(c, k)$ API allows a smart contract to permanently remove a specific key from both the current private state and all historical versions stored in the private data store. This is distinct from \texttt{DelPrivateData}, which only removes the key from the current state while retaining historical write-sets. When a purge is committed, the peer records a \textit{purge marker} keyed by $(c, H(k))$ that permanently indicates the key has been purged at a given block and transaction index. A \textit{hashed index} maps each hashed key to the block and transaction locations of its historical write-sets, enabling efficient background cleanup without scanning the entire ledger state.

Purge interacts with the reconciliation protocol: when a peer responds to a reconciliation or reactive retrieval request, it filters out keys that have been purged locally, returning a partial write-set rather than a complete one. The requesting peer accepts partial write-sets by verifying individual key hashes rather than a single hash over the entire collection write-set. If a requesting peer receives data for a key it has already purged, it discards the data. Conversely, if a peer requests data that the responder has already purged, the responder omits those keys and the requester re-queues the request until it observes the purge transaction in a committed block, at which point it removes the entry from its missing data set.
In both mechanisms, on-chain hashes are preserved after private data removal, keeping the audit trail tamper-evident; the implications are discussed in \S\ref{s:threat-model}.
