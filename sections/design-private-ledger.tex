\subsection{Transaction Simulation}
Similar to the original protocol, the simulation phase involves simulation of transaction
and collection of read-write set. In addition to that, the proposed protocol introduces
a few concepts such as transient data, hashed \& private read-write set, transient store,
and private data dissemination as described next.
\par
\textbf{Transient Data in Transaction Proposal.}
Each transaction in the block contains the client's input to the smart contract execution, for auditing.
However, private data as an input in the transaction might leak the sensitive data if it appears as-is in the block.
Encrypting this secret information at the client side and decrypting it in the smart contract, is problematic because
\begin{itemize}
	\item Using symmetric encryption induces either chicken-and-egg problem or the need to send these keys separately,
        complicating the interaction between clients and peers.
	\item It is tempting to use asymmetric encryption and have the client encrypt the secrets using the public key of the peers.
        This requires the client to encrypt the data separately for each peer that is involved in the smart contract execution,
        however requires infrastructure for public key discovery.
\end{itemize}
Instead, the client passes the secret information to the peer in a different data section, denoted as \emph{transient data}
than the public input to the smart contract computation which is omitted from the transaction that ends up in the blockchain.
In order to preserve some minimal level of auditing capabilities, the peer includes the hash of all data sent from the client
(including the transient data) in the input to the signature over the smart contract computation results.
This allows a client to prove to an auditor that the computation was either done correctly or incorrectly by knowing the
transient data.
\par
\textbf{Hashed Read-Write Set and Private Write Sets}
\par
While simulating a transaction involving private data, node constructs (1) a public data read-write set,
(2) a hashed data read-write set, and (3) one or more private data write set. As mentioned in Table~\ref{table:apis},
the usage of one or more of \texttt{GetState(), PutState(), and DelState()} in the smart-contract results in the
constructions of public data read-write set. Similarly, the usage of one or more of \texttt{GetPrivateData()},
\texttt{PutPrivateData()}, \texttt{DelPrivateData()}, \texttt{GetPrivateDataHash()}, results in the constructions of the
hashed data read-write set.
As discussed in \S\ref{}, the hashed data read-write set is required such that all nodes in the blockchain network can perform
the optimistic concurrency check to ensure serializable isolation irrespective of collections' accessed/modified and
the respective membership.
Further,
a private write-set per collection is constructed when either \texttt{PutPrivateData()} or \texttt{DelPrivateData()} or
both are used. In other words, a private write-set is constructed for each private data collection that the transaction modifies.
As each private-write set should be visible only to the respective collection members, node packages only the public and hashed
data read-write as a simulation result while storing the private data write-sets in the node's transient store. Further, the
node share each private write-set associated with the transaction with other nodes that are member of the corresponding private data
collection through private data dissemination as discussed later. This is also a reason for creating one private data write-set
per collection.

\subsection{Possible attacks and their mitigations}
\par
The two types of actors in our protocol are clients and peers. We analyze how our design copes with the presense of malicious actors of both types.

The role of a client is to collect the responses from the peers, containing signed simulation results, bundle them into a transaction, and finally send it to the ordering service.

The role of a peer is to mark private data collection updates with corresponding hashes, and disseminate hash pre-images to eligible peers.

\textbf{Storage space overload}:  Recall, before reception of hashed writes in a block, the hash pre-images are saved in a temporary storage location called the transient store. Upon reception of a hashed write and locating the corresponding pre-images either from the transient store or from other peers, the peer purges the bespoken pre-images from the transient store. However, a malicious or faulty client may crash before it gets the chance to send the transaction to the ordering service, thus the corresponding hashed writes will never trigger deletion of pre-images stored in the transient store at the transaction simulation phase. Repeating this pattern over time will lead to excess storage accumulation which will also slow down the lookup speed of the transient store because of space increase. 

To that end, we introduce a \emph{Retention period} rule to the transient store, which automatically purges hash pre-images associated to blocks too far in the past. More specifically, whenever a peer simulates a transaction, it records its ledger height (henceforth \emph{simulation height}) as part of the set of pre-images produced by the simulation of that transaction. When the peer's ledger height advances further than the retention period without encountering  corresponding hashes, it automatically purges these pre-images from the transient store. 

A natural question that rises is what about disseminated pre-images that take place before the ordering? After all, peers also disseminate the pre-images to other peers in the transaction simulation phase for redundancy. We take a simple yet effective approach and include the simulation height along the pre-images that are disseminated. To protect against disseminating peers maliciously selecting an extremely high simulation height (such as $2^{64} - 1$), the receiving peers refuse if their ledger height is too behind the claimed simulation height of the disseminating peer.

\textbf{Throughput degradation}: Besides malevolently increasing the storage space of the transient store, a malicious client may attempt to send a transaction with a fabricated simulation which was never signed by sufficient peers to satisfy the endorsement policy. As a result, peers would try to seek out pre-images among their neighbors until giving up due to reaching a maximum attempt threshold. Since peers can only handle a single block at a time, the throughput would greatly degrade due to the time spent looking for pre-images that are nowhere to be found.

Consequently, we perform endorsement policy validation \emph{before} we proceed pulling the pre-images. This strategy ensures that every hash write in a transaction was simulated and signed by a threshold of peers, and therefore malicious clients (or colluding peers that do not satisfy the agreed upon endorsement policy) cannot corece peers to seek out pre-images that cannot be obtained because the transactions will be deemed invalid and pulling their pre-images will be skipped, thus mitigating the attack.
