\section{Proposing Private Data Collections For Data Privacy}
\label{pdc}
A naive approach to achieving data privacy involves using an external, off-chain data store (e.g., a traditional database) to house private data while only storing its cryptographic hash on the blockchain. However, this decoupled approach is insufficient for enterprise applications as it breaks the fundamental trust and consistency models of the blockchain. First, it cannot guarantee transactional atomicity: in a food supply chain, an update to a public shipment record and its private invoice must be atomic, yet achieving atomicity via two-phase commit across a decentralized blockchain and an external database is notoriously fragile. Second, it lacks integrated smart-contract access control: an integrated solution allows the same smart contract that enforces business logic to also gate access to the private data---ensuring, for example, that private medical records are only accessible to authorized practitioners. Finally, dynamic membership becomes a manual, error-prone process: if a new organization joins a consortium, they must be manually granted access to historical external data, whereas an integrated solution allows blockchain nodes to automatically synchronize private state according to the collection policy. By managing private data directly within the blockchain nodes, we ensure that the private data inherits the same availability, consistency, and security properties as the replicated ledger. To achieve this, we propose an integrated \textit{private data collection} mechanism.

\subsection{Key Idea}
Our proposal achieves data privacy by decoupling the dissemination and storage of private data from the maintenance of the shared, immutable transaction ledger. To hide private data from specific members in a blockchain network, we propose two complementary mechanisms: (1) \textit{decoupled storage}, where private data is shared and stored only on nodes owned by authorized members via a private data \textit{collection} with defined membership; and (2) \textit{on-chain evidence}, where the cryptographic hash of the private data is stored on all nodes as part of the replicated ledger for tamper detection.

Formally, a private data collection $\mathcal{C}$ is defined as a tuple:
\begin{equation*}
\mathcal{C} = (id, \mathcal{M}, \mathcal{E}, n_{dissem}, n_{required}, \tau)
\end{equation*}
where $id$ is a unique collection identifier, $\mathcal{M}$ is the membership policy specifying the set of authorized organizations, $\mathcal{E}$ is the endorsement policy governing write access, $n_{dissem}$ is the maximum number of member organizations targeted during proactive dissemination, $n_{required}$ is the minimum number of member organizations that must acknowledge receipt before a transaction can proceed, and $\tau$ is the block-to-live (BTL) retention parameter specifying the number of blocks for which private data remains in the private data store after being committed (where $\tau = 0$ signifies indefinite retention). A smart contract may define multiple collections, each with independent policies. Note that the collection's endorsement policy $\mathcal{E}$ may differ from the smart contract's own endorsement policy, which governs writes to the public state shared across all members. This is because the collection membership $\mathcal{M}$ is typically a subset of the network participants, requiring a distinct endorsement policy drawn from its authorized members.

Figure~\ref{fig:fsc-pvt-data-food-traceability} showcases the use of private data collections in the food traceability use-case. Five collections are defined based on participant relationships; the replicated ledger contains public data (e.g., order IDs, product lots) and the hash of each private data element, while private data (e.g., invoices, payments) are stored in the appropriate collection. This ensures that competitors can verify product provenance without gaining access to private business contracts. The same mechanism extends to personal data management: a hospital stores a patient's medical records in a collection where its node is a member, and when needed---such as for an insurance claim---the record can be copied to a collection where the insurance provider is a member. Because the record's hash is present in the replicated ledger, the receiving party can independently verify its authenticity, resolving the conflict between ledger transparency and patient privacy.

\subsection{Design Requirements}
The use cases in \S\ref{motivation} and the architectural constraints of our key idea give rise to seven design requirements, organized into three categories.

\smallskip
\noindent\textit{Correctness.} A private data mechanism must preserve the transactional guarantees of the underlying blockchain. In the food supply chain, updating a public shipment record and its private invoice must be atomic---a partial update would leave the ledger in an inconsistent state. Similarly, the on-chain hash must faithfully reflect the off-chain data so that any party can detect tampering.
\begin{enumerate}[topsep=0pt,noitemsep,leftmargin=15pt]
		\item[\textbf{R1}] \textit{Transactional Atomicity and Isolation:} When a transaction modifies both public and private data, the system must guarantee atomicity and serializable isolation to maintain overall consistency.
        \item[\textbf{R2}] \textit{Hash-Data Integrity:} The hash of the private data stored on the replicated ledger must always remain in sync with the actual private data stored in the off-chain collection, acting as a verifiable proof of existence.
\end{enumerate}

\noindent\textit{Access Control.} Privacy is only meaningful if the system enforces who can read and write private data. In healthcare, only authorized practitioners should access patient records; the same smart contract that enforces clinical business logic should also gate access to the underlying data. Furthermore, private data must never be processed on unauthorized nodes, even transiently during execution.
\begin{enumerate}[topsep=0pt,noitemsep,leftmargin=15pt]
        \item[\textbf{R3}] \textit{Smart-Contract Controlled Access:} Private data in a collection must only be accessible or modifiable via a smart-contract, ensuring all data interactions adhere to agreed-upon business processes.
        \item[\textbf{R4}] \textit{Authorized Execution Only:} Transactions involving private data must only be executed on nodes that are members of the specified collections, preventing unauthorized nodes from seeing or processing private data.
\end{enumerate}

\noindent\textit{Operational.} Enterprise consortia evolve over time: new organizations join, existing members leave, and regulations impose data retention limits. In global trade, a new carrier joining GSBN must synchronize historical shipping documents; under GDPR, a patient may exercise the ``right to forget,'' requiring purging of personal data. Because private data is excluded from the consensus-driven block delivery, the system must also guarantee that authorized nodes can obtain it through an alternative mechanism.
\begin{enumerate}[topsep=0pt,noitemsep,leftmargin=15pt]
		\item[\textbf{R5}] \textit{Dynamic Membership:} Collection membership must be mutable. New members must be able to synchronize historical private data, while removed members should lose access to any data persisted after their removal.
		\item[\textbf{R6}] \textit{Data Lifecycle Management:} The system must support a block-to-live (BTL) property for private data, allowing for automatic purging to comply with data retention policies or ``right to forget'' regulations.
		\item[\textbf{R7}] \textit{Data Availability:} Since private data is excluded from the consensus-driven block delivery, the system must ensure that all authorized nodes can obtain the private data they need before committing a block, even under network partitions and peer failures.
\end{enumerate}

\subsection{Implication of Blockchain Replication Protocols}
To realize the requirements outlined in \S\ref{pdc}, we must identify a blockchain architecture that natively supports decoupled execution and selective data dissemination. In a permissioned blockchain network, a \textit{client} submits transactions on behalf of an application; a \textit{peer} is a node owned by a member organization that maintains a copy of the ledger, executes smart contracts, and validates transactions; and an \textit{ordering service} is a decentralized cluster of nodes---contributed by the same participating organizations---that establishes the total order of transactions via a consensus protocol. Each peer belongs to exactly one organization, and organizations may operate multiple peers for redundancy. We evaluate three prevalent replication protocols, depicted in Figure~\ref{fig:replication-protocol}, to determine their suitability as a foundation for our mechanism. Throughout this analysis, we assume a crash-fault tolerant (CFT) model for authorized peers; the full threat model, including Byzantine considerations for the ordering service, is presented in \S\ref{s:threat-model}.

\textit{Order-Execute-Commit and Execute-Order-In-Parallel \& Commit.}
These two protocols---shown in Figures~\ref{fig:replication-protocol}(a) and (b)---differ in whether transactions are ordered before or after execution, but share a fundamental limitation: every node must execute (or re-execute) every transaction to maintain ledger consistency. The order-execute-commit protocol (Ethereum~\cite{ethereum}, Quorum~\cite{Quorum}, Hyperledger Besu~\cite{besu}) orders first, then broadcasts blocks for serial execution on all nodes. The execute-order-in-parallel protocol (BlockchainRDBMS~\cite{pgchain}) executes first, then orders, but still requires all nodes to verify the final block. In both cases, restricting execution to a subset of nodes would break the state machine replication (SMR) guarantees, violating R4 (Authorized Execution Only). Since all nodes execute every smart contract, private data accessed during execution is exposed to the entire network (violating R3), and the absence of a separate validation phase means there is no way to verify private transactions without re-executing them (violating R7).

\textit{Execute-Order-Validate-Commit.}
Figure~\ref{fig:replication-protocol}(c) presents four phases: simulation (steps 1--2), ordering (steps 3--5), validation (step 6), and commit (step 7). A transaction is speculatively executed on a subset of nodes according to a specific endorsement policy; the results (read-write sets) are then ordered by a consensus service; finally, all nodes perform validation (e.g., policy and serializability checks) before committing valid transactions. Hyperledger Fabric~\cite{hf} implements this protocol. This architecture is uniquely suited for our mechanism because it can satisfy all seven requirements: transactions are only simulated on a subset of nodes, satisfying R4 and R3 by routing private data proposals only to authorized collection members; the public validation phase ensures hashes are correctly recorded on the replicated ledger (R2); OCC-based validation guarantees atomicity and isolation across both public and private state (R1); the decoupling of simulation from consensus enables a peer-to-peer dissemination layer supporting dynamic membership (R5) and ensuring data availability for authorized nodes (R7); and because private data is stored separately from the immutable ledger, lifecycle management (R6) can be realized by purging expired data while retaining on-chain hashes.

Based on this analysis, we adopt the execute-order-validate-commit protocol as the foundation for our design. The subsequent sections describe the generic design of our private data collection mechanism.
