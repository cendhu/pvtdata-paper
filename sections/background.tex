\subsection{Architecture of Hyperledger Fabric}
\label{hf}
Hyperledger Fabric consists of three entities---client,
peer, and orderer. The transaction flow in Fabric
involves all three entities and composes of four phases---
simulation, ordering, validation, and commit, as depicted
in Figure~\ref{fig:txflow}. Figure~\ref{fig:peer} shows the various components in a
peer, which involves in the execution of simulation, validation, and commit phase. The
communication between different entities happens via google
remote procedure call (gRPC)~\cite{grpc}.
% Note that the RPC is used between
% the peer and smart-contracts to support contract written in
% multiple languages such as nodejs~\cite{nodejs}, java~\cite{java}, and
% golang~\cite{golang}.
\par
\textbf{Phase 1---Simulation.} A client submits a transaction proposal to a peer to invoke a
smart-contract which implements the transaction logic. %On receiving a transaction proposal,
The \textit{endorser} validates
and passes the input present in the proposal message to the
appropriate \textit{smart-contract}. Depending on the input,
the \textit{smart-contract} executes the logic and reads/writes the states it
manages by issuing \texttt{\small{GetState()}}, \texttt{\small{PutState()}}, \&
\texttt{\small{GetStateByRange()}} calls back to the \textit{endorser}. For a read
request, i.e., \texttt{\small{GetState()}}, the \textit{endorser} reads the state
from the \textit{state DB} and adds it to the read set of the transaction maintained
at the \textit{endorser} before passing it to the \textit{smart-contract}.
For a write request, i.e., \texttt{\small{PutState()}}, the \textit{endorser} stores
the value in the write set of the transaction without modifying the
\textit{state DB}. For a range read query, i.e., \texttt{\small{GetStateByRange()}}
with a \textit{start} and \textit{end} key, the \textit{endorser} returns an iterator to
the smart-contract. By calling the \texttt{\small{Next()}} on the iterator, the smart-contract
reads states. Instead of storing these states in the read set, the \textit{endorser}
stores the \textit{start} key, \textit{end} key, and a list of keys read in a separate
data structure called \textit{range query info}. Once the transaction execution completes,
the \textit{endorser} cryptographically signs/endorse the transaction response
(which includes the transaction proposal, read-write set, and range query info)
before sending it to the client.
\par
The client can submit the
transaction proposal to multiple peers simultaneously depending upon the
endorsement policy~\cite{endorsement1}
defined for the \textit{smart-contract}. For example, the \texttt{OR}(\textit{OrgA.peer},
\texttt{AND}(\textit{OrgB.peer}, \textit{OrgC.peer})) policy requires either a signature
from organization A's peer or two signatures, i.e., from organization B's and organization C's peer.
Each \textit{smart-contract} maintains its states. One \textit{smart-contract}
can access/modify states maintained by another contract by invoking it. A
\textit{smart-contract} invoking another enables the application designer
to split a monolithic contract into multiple micro contracts,
each performing a specific task.

\textbf{Phase 2---Ordering.} 
%Once a client collects the required endorsements for the transaction,
The client submits the endorsed transaction response to the ordering
service. An ordering service, which consists of orderer nodes from different
organizations, employs consensus protocol~\cite{raft} to order the received
transactions and create a block. Each block has a sequence number called
\textit{block number}, the hash of the previous block, the hash of the current block,
a list of ordered transactions (i.e., commit order), and the orderer's
signature. The orderer nodes broadcast the created block to peers.
\par
\textbf{Phase 3---Validation.} The \textit{gossip} component of the peer
receives blocks and stores them in a \textit{block queue}. The peer 
verifies the following two conditions per transaction in a block before committing
them: (1) collected
endorsements satisfies the endorsement policy specified for the smart-contract;
(2) transaction is serializable. The \textit{endorsement policy validator} verifies
the endorsements against the endorsement policy by checking whether the set of
signatures on the transaction response is signed by the set of organizations
specified in the endorsement policy. If a transaction invoked multiple smart-contracts
(via a feature that allows a smart-contract to invoke another),
the policy of each of the smart-contract must be satisfied. A transaction is marked
invalid when it does not have adequate endorsements.
\par
Once the \textit{policy validator} validates all transactions, the \textit{serializability validator} applies optimistic
concurrency control (OCC)~\cite{occ} using the read-write set present in the
transaction response of valid transactions. To facilitate OCC, the Fabric adds a version identifier
to each state stored in the blockchain. The version for a state is nothing but
a combination of a block number and the transaction number within the block
which last updated/created the state.
All the validator checks for is if the state that a transaction has read, to decide on
its write-set, have not been modified by preceeding valid transactions in
the block and already committed transactions. Further, the validator re-executes
range queries present in the \textit{range query info} to detect any phantom read
as it violates serializability property.
% \par
Note that the \textit{policy validator} parallelly validates transactions
in a block whereas the \textit{serializability validator} serially validates each
transaction %in the order they appear in the block
such that it can take into account of write-set of previous valid transactions within the block while executing OCC.

\par
\textbf{Phase 4---Commit.} After the validation phase,
first, the \textit{committer} stores the block in the \textit{block
store}, which is a chain of blocks stored in a file system along with the
transaction validity information. Second, the \textit{committer} applies the write
set of all valid transactions to the \textit{state database}, which maintains
all active states. Third, it stores all valid and invalid transactions' write set to the
\textit{history database}, which maintains both active and inactive states. The disk
writes to \textit{block store}, \textit{state DB}, and \textit{history DB} are serial and
synchronous to handle recovery after a failure. Note that the \textit{history DB} only
maintains an index in the form of state to a list of \{\textit{block number},
\textit{transaction number} within the block\} such that the actual history of each
state can be retrieved from the \textit{block store} using the transaction's write-set.

\subsubsection{Use Case Study 1: Food Traceability}
In September 2006, there was an outbreak of food-borne illness caused by Escherichia coli
bacteria~\cite{ecoil} in 26 U.S. states. This outbreak was linked to bagged spinach. By October 6, 2006, 199 people had been
infected, including three people who died and 31 who suffered kidney failure. It took regulators two
weeks to conduct the backtrace and determine the exact source of the outbreak. During those two weeks,
many people got sick, and tons of good spinach was unnecessarily discarded---wrongfully
wasted because we could not tell the good spinach from the bad~\cite{blockchain-for-business}.
An efficient food traceability mechanism is the key to reduce the backtrace time in such a scenario.
Traceability is the ability to track and record the movement of food products throughout a supply chain, which
includes growing, harvesting, packing, manufacturing, repacking, storage, distribution, and sale to the consumer.
With this information, it is possible to trace backward in the supply chain (usually to find the source of a
problem) as well as trace forward (usually to do a recall of contaminated products).

Food Standards Australia New Zealand (FSANZ)~\cite{fs}, European Commission's food safety policy~\cite{eu-foodsafety},
and Food and Agriculture Organization of United Nations (FAO)~\cite{fao-traceability} provide various guidelines on what
records need be kept in order to enable food traceability. The following are some of the records:
\begin{enumerate}[topsep=0pt,noitemsep,leftmargin=15pt]
	\item name and address (and other contact details) of farmers/suppliers and a description of products or inputs supplied.
	\item the name and contact information of the logistics providers and a description of products transported.
	\item name and addresses (and other contact details) of customers and a description of the product supplied to them along with the product ID.
    \item date of transaction or delivery.
    \item batch or lot identification (or other markings such as product ID).
    \item volume or quantity of product supplied or received.
	\item raw materials used.
	\item conservation and storage methods for the product.
\end{enumerate}
As a result, when an outbreak of food-borne illness occurs, it would be possible to traceback from the customer
to the product ID to the supplier/farmer. Similarly, a recall can be performed by tracing forward from the farmer/supplier
to the product lot to customers. If these records are kept in the form of papers or in a centralized database maintained at
individual operators, during an outbreak, a lot of communication needs to happen between various food business operators
to perform traceback or trace forward. Such a method is very inefficient, time-consuming, and expensive.
Hence, a blockchain-based food traceability
solution is needed for food safety. Further, such a solution helps to improve the overall supply chain efficiency, to
allow us to easily communicate information that customers are asking for (i.e., fork to farm), to easily handle disputes, 
and to help autheticate claims made in a product. With blockchain, network members can track the food products as they
travel from farm to fork. Recently, Walmart, a retailer, did an experiment that traced the origin of sliced mangos from
Walmart stores back to the farm. This process showed a radical improvement from approximately seven days it took to
conduct the backtrace by using traditional methods down to 2.2 seconds~\cite{7-to-2, walmart-hf} using Hyperledger Fabric.
\par
Figure~\ref{fig:fsc} shows actors involved in a simplified chocolate bars supply chain. For manufacturing a
chocolate bar, cacao and milk are required (assume a bar of sugar-free chocolate). Usually, manufacturers
buy raw materials from suppliers rather than directly contacting various farmers. Once chocolate bars are produced,
via distributors, it would be sent to various retailers across the globe so that the chocolate bars can be sold
to customers. Note that distribution involves import/export authorities, customs, and other entities. For simplicity,
we do not cover all involved actors.
Figure~\ref{fig:tx-flow-fsc} shows a sample business process in a chocolate supply chain. The process involves
order placement, invoice creation, shipping arrangement, delivery, and proof of payment. The whole process
can be implemented using a permissioned blockchain platform with the help of smart-contracts. Given that the
transaction details are stored in a replicated ledger, it helps in resolving disputes, and also it enables
tamper-proof record management for food traceability. Note that blockchain also helps in standardizing the
business process and data format through smart-contract.

\subsubsection{Use Case Study 2: Patient-Centric Health-Care}
In health-care, a practitioner does not have access to all medical records of a patient. This is because the Electronic
Medical Record (EMR) systems assume that a patient sees only practitioners in one hospital. However, the reality is that
on average a patient in the US sees 18.7 different practitioners/hospitals during his/her lifetime~\cite{}. As existing EMR systems are
hospital-centric, there is no interest to interoperate or share data with other hospitals due to competitions. As a result,
the patient's EMR are trapped in different islands, which creates a hurdle to execute personalized health-care. Further, a hospital-centric
delivery model using the current framework of existing technologies results in a lack of transparency and trust. A
practitioner can provide a wrong treatment and later tamper the EMR to hide his mistakes, given that the EMR system is managed
at his/her hospital. Further, the Health Insurance Portability and Accountability Act (HIPAA) states that patients should own
their medical records and control access permission.
\par
Another issue in health-care is related to insurance. There is a growing concern in the insurance industry on increasing number of health-care
insurance frauds. The FBI estimates that health care fraud costs American taxpayers \$80 billion a year~\cite{}. The following
are the commonly committed frauds: (1) duplicate and inflated bills; (2) manipulating pre-policy health check-up records; (3) inflated billing,
billing for services not provided; (4) unwarranted procedures, excessive investigations, expensive medicines;
(5) over utilisation, extended length of stay; (6) fudging records, patient history to name a few.
\par
To resolve the problems mentioned above, we need a blockchain-based solution, as shown in Figure~\ref{fig:health-care}. Here, the
patient owns their EMR and controls access permission. Hospitals can create EMR for patients as a result of some medication or treatment.
As the information is shared using a replicated and immutable ledger, it adds more transparency \& trust, enables sharing of
EMR records, and can reduce the number of fraudulent events in the insurance industries. Further, the smart-contract can help in
standardizing the EMR data format.
\par
\textbf{Data Privacy Violation.} \textit{Sharing of sensitive data
		such as EMR of a patient in a replicated ledger is usually unacceptable as it leaks private information about
		a patient though there is some access control. Further, there are many laws, including HIPAA privacy rule~\cite{},
		GDPR~\cite{}, which ask the hospital to protect the patient's privacy.}
