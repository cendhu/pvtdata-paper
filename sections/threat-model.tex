\section{Threat Model and Security Analysis}
\label{s:threat-model}
Our design assumes a decentralized network of organizations in which some nodes may be faulty or malicious. We characterize the trust assumptions for each entity class and the security implications that follow.

\textit{Peer Nodes.} For \textit{correctness}, the endorsement policy is the trust boundary for hash-data integrity (R2). Each endorsing peer independently computes the hash of private data during simulation and signs the result; since the endorsement policy requires matching responses from peers across multiple organizations, no single compromised peer can record a fraudulent hash or unilaterally influence transaction outcomes. However, if all endorsing members of a collection collude, they can commit a hash that does not correspond to the actual private data, breaking R2. This is an inherent limitation: the system guarantees hash-data integrity only under the assumption that the endorsement policy holds---i.e., at least one endorser is honest. \textit{Confidentiality} is more nuanced. A malicious peer can leak private data it legitimately holds, or refuse to disseminate data to eligible peers; these are inherent limitations of access-control-based privacy, which prevents unauthorized access but cannot constrain how an authorized party uses data it possesses. Confidentiality is therefore bounded by the weakest collection member. Purging offers no stronger assurance: a malicious member can retain private data locally without honoring purge requests, so purge is best-effort rather than a cryptographic guarantee.

\textit{Clients.} Clients may be malicious, submitting inconsistent private data to different endorsers, fabricating simulation results without sufficient endorsements, or flooding the system with orphan write-sets. The hash mechanism neutralizes inconsistent submissions: if a client sends different private data to different endorsers, each endorser computes a different hash and includes it in its signed response; because the endorsement policy requires matching responses from multiple endorsers, the divergent hashes render the endorsements incompatible and the transaction is rejected during validation. Fabricated simulations are similarly neutralized---peers perform endorsement policy validation \textit{before} attempting to pull any missing pre-images, discarding invalid transactions without fetching their data. For write-set flooding, where a client simulates many transactions but never submits them, a retention window $\Delta$ bounds how long orphan entries persist: the transient store automatically purges entries whose simulation height lags the current ledger height by more than $\Delta$ blocks, and receiving peers reject disseminated write-sets outside this window. The same retention window defends against a malicious \textit{peer} flooding fake write-sets to other collection members; such activity is tracked through monitoring metrics that allow operators to identify and remediate faulty peers.

\textit{Other Considerations.} Only public and hashed read-write sets are submitted to the ordering service, so ordering nodes never see private data payloads; the ordering layer supports both CFT (Raft) and BFT consensus. We assume a partially synchronous network in which partitions are possible but eventually resolve---a partition may delay private data dissemination, but a peer that cannot obtain private data before commit records it as missing and reconciles once connectivity is restored. Private data must be salted by the client before hashing to prevent adversaries from predicting or confirming the underlying data through brute-force hash comparison. Regarding data lifecycle, purging removes private data from the off-chain store but the on-chain hash is retained on the immutable ledger; this may not fully satisfy regulations such as GDPR that require complete erasure, and we expect peers to be honest in executing purge operations. Finally, private data collections protect payload content but not transaction \textit{metadata}: the hashed read-write sets recorded on the public ledger reference specific collections, and collection membership policies are visible to all network participants, so any peer can infer which organizations are party to a private transaction even without seeing the underlying data. Mitigating this leakage would require techniques such as zero-knowledge set membership proofs~\cite{zk-set-membership} or anonymous credentials~\cite{camenisch-lysyanskaya}.
