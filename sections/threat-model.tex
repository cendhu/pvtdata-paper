\section{Threat Model and Security Analysis}
\label{s:threat-model}
Our design assumes a decentralized network of organizations in which some nodes may be faulty or malicious. We characterize the trust assumptions for each entity class and the security implications that follow.

\textit{Peer Nodes.} We assume a crash-fault tolerant (CFT) model for authorized peers. For \textit{correctness}, the endorsement policy is the trust boundary for hash-data integrity (R2). Each endorsing peer independently computes the hash of private data during simulation; since the endorsement policy requires matching responses from multiple organizations, a single compromised peer cannot record a fraudulent hash. However, if all endorsing members of a collection collude, they can commit a hash that does not correspond to the actual private data, breaking R2. This is an inherent limitation: the system guarantees hash-data integrity only under the assumption that the endorsement policy holds---i.e., at least one endorser is honest. \textit{Confidentiality} is more nuanced. A malicious peer can leak private data it legitimately holds, or refuse to disseminate data to eligible peers; these are inherent limitations of access-control-based privacy, which prevents unauthorized access but cannot constrain how an authorized party uses data it possesses. Confidentiality is therefore bounded by the weakest collection member. Purging offers no stronger assurance: a malicious member can retain private data locally without honoring purge requests, so purge is best-effort rather than a cryptographic guarantee.

\textit{Ordering Service.} Only public and hashed read-write sets are submitted to the ordering service, so ordering nodes never see private data payloads. The ordering layer supports both CFT ordering via Raft, which tolerates crash failures as long as a majority of nodes remain available, and BFT ordering, which further tolerates Byzantine faults provided fewer than one-third of ordering nodes are malicious.

\textit{Clients.} Clients may be malicious, submitting inconsistent private data to different endorsers or forging simulation results. The hash mechanism neutralizes the first attack: if a client sends different private data to different endorsers, each endorser computes a different hash and includes it in its signed response. Because the endorsement policy requires matching responses from multiple endorsers, the divergent hashes render the endorsements incompatible and the transaction is rejected during validation.

A subtler attack is write-set flooding: a client simulates many transactions but never submits them, causing orphan write-sets to accumulate in the transient store. We counter this with a retention window $\Delta$---the transient store automatically purges entries whose simulation height lags the current ledger height by more than $\Delta$ blocks, and receiving peers reject disseminated write-sets outside this window. Setting $\Delta$ requires care: it must exceed the expected block count between simulation and commit, which depends on network latency and block creation rate; too small a value risks permanently losing private data for valid transactions delayed in the ordering pipeline. In our implementation, $\Delta$ is governed by \texttt{transientstoreMaxBlockRetention} (default: 1000 blocks), and \texttt{pullRetryThreshold} (default: 60\,s) bounds how long a peer retries reactive retrieval before recording data as permanently missing. A second attack vector involves fabricated simulations lacking sufficient endorsements; peers counter this by performing endorsement policy validation \textit{before} attempting to pull any missing pre-images, discarding invalid transactions without fetching their data.

\textit{Network.} We assume a partially synchronous network in which partitions are possible but eventually resolve. A partition may delay private data dissemination, but correctness is not compromised: a peer that cannot obtain private data before commit records it as missing and reconciles once connectivity is restored.

\textit{Metadata Leakage.} Private data collections protect payload content but not transaction \textit{metadata}. The hashed read-write sets recorded on the public ledger reference specific collections, and collection membership policies are visible to all network participants; any peer can therefore infer which organizations are party to a private transaction, even without seeing the underlying data. Eliminating this leakage would require cryptographic techniques such as zero-knowledge membership proofs~\cite{?} or anonymous credentials~\cite{?}, which let a peer prove membership in an authorized collection without disclosing which collection it belongs to. We leave such extensions as future work.
