\section{Implementation}

The private data collection mechanism is designed for any blockchain platform that follows the simulate-order-validate-commit replication protocol. We implemented it within Hyperledger Fabric~\cite{hf}, comprising approximately 33,000 lines of Go code (production logic and tests), shipped in production since v1.2. Figure~\ref{fig:peer-impl} shows how the new components (gray) extend the existing peer architecture (white).

\textit{Private Data Flow.}
During simulation, the \textit{endorser} writes private write-sets to the \textit{transient store}~(\ding{182}), a temporary staging area keyed by transaction ID, while the \textit{Pvtdata Dissemination} component concurrently pushes private write-sets to authorized collection members~(\ding{183}). When the \textit{gossip} layer delivers an ordered block from the ordering service into the \textit{block queue}~(\ding{184}), the \textit{endorsement policy validator} checks each transaction against both chaincode-level and collection-level policies~(\ding{185}). For valid transactions, the peer retrieves private data from its local \textit{transient store}~(\ding{186}); any data still missing---due to network delays---is pulled from eligible peers by the \textit{Missing Pvtdata Reconciler}~(\ding{187}). The \textit{serializability validator} then performs blind OCC validation against the public and hashed state partition in the \textit{State DB}, verifying serializability without accessing private data (Theorem~1)~(\ding{188}). Finally, the \textit{committer} moves validated private data from the transient store to the durable \textit{Private Data Store}~(\ding{189}), which also tracks eligible and ineligible missing data for future reconciliation, and updates the public, hashed, and private partitions in the \textit{State DB}~(\ding{190}), where per-chaincode namespace isolation prevents cross-contract access to hashed or private state. A background \textit{Purge DB} process expires private data based on the block-to-live parameter $\tau$, and the \texttt{PurgePrivateData} API (Table~\ref{table:apis}) supports application-controlled deletion by committing purge markers that prevent data from appearing in queries or being reconciled, providing the ``right to forget'' behavior required by data protection regulations.

\textit{Commit-Path Optimizations.}
Initial microbenchmarks of the commit path revealed a 50\% throughput penalty for PDCs (720 vs.\ 1440 tx/s). Profiling identified cryptographic membership checks and serial disk I/O as the dominant bottlenecks. We applied a series of crash-safe optimizations (Table~\ref{table:optimizations}) that reduced the commit-path overhead to just 6\%; end-to-end overhead across industry workloads is 3--14\% (\S\ref{s:eval}). Key techniques include replacing the commit-time collection membership check---which originally invoked the full policy evaluation engine with certificate deserialization and signature verification---with an O(1) organization identifier lookup (safe because cryptographic policy enforcement already occurs at endorsement and validation time), parallelizing block-store and private-data-store commits, deferring transient-store purge to a background routine, and adding a transient-store read cache.
