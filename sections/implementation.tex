\section{Implementation}

The private data collection mechanism described in \S\ref{s:design} was built directly into Hyperledger Fabric~\cite{hf}, an open-source permissioned blockchain platform, mapping each architectural component to a peer-internal service.

\textit{Fabric Architecture Overview.}
Fabric's simulate-order-validate-commit paradigm is described in \S\ref{pdc}. Within a peer, the central components are the \textit{endorser} responsible for chaincode simulation, the \textit{gossip} layer for block dissemination, endorsement policy and serializability validators, and a \textit{committer} that persists valid transactions to the \textit{block store}, \textit{state DB}, and \textit{history DB}. Figure~\ref{fig:peer-impl} shows how private data collection components (gray) extend the existing architecture (white). During endorsement, the \textit{endorser} stores private data in the \textit{transient store} and disseminates it to other authorized members via the gossip-based private data dissemination protocol. When the \textit{gossip} layer receives an ordered block, the endorsement policy validator checks the block's transactions; for each valid transaction, the peer fetches the corresponding private data from its local transient store or, if missing, pulls it from other eligible peers. The retrieved private data then moves to the serializability validator and finally to the \textit{committer}, which persists it alongside the block. Any entries still missing after the commit window are fetched asynchronously by a separate \textit{reconciler} within the gossip layer.

\textit{Storage and State Management.}
The four logical stores described in \S\ref{s:design}---transient store, private data store, eligible missing data tracker, and ineligible missing data tracker---are each backed by a LevelDB instance embedded within the peer process. The partitioned state management scheme (\S\ref{s:design}) is realized by co-locating public, hashed, and private state in the same physical State DB (LevelDB or CouchDB, operator-selectable) and distinguishing them via internal key prefixes; the peer's data management layer enforces namespace isolation so that no smart contract can access another chaincode's hashed or private partitions. The hash function $H$ is configurable per channel via a configuration transaction; Fabric supports SHA-256, SHA-512, and other SHA-family algorithms, with SHA-256 as the default.

\textit{Dissemination, Membership, and Lifecycle.}
The proactive dissemination and reactive retrieval protocols (\S\ref{s:gossip}) are realized as new gRPC-based message types within Fabric's gossip component; a configurable retry mechanism with exponential back-off handles transient failures in both paths. Collection policies are stored as part of the chaincode definition in the channel configuration and evaluated at validation time as described in \S\ref{s:design}. Two complementary purging mechanisms implement the data lifecycle management policy (\S\ref{s:design}): a background retention process within the \texttt{PvtDataStore} automatically expires private data based on the block-to-live parameter $\tau$, running in asynchronous batches so as not to impede commit throughput; for application-controlled deletion, the \texttt{PurgePrivateData} API (Table~\ref{table:apis}) commits a purge marker to the ledger, and once committed, the marker prevents the data from appearing in query results or being reconciled from remote peers, providing the ``right to forget'' behavior required by data protection regulations.

\textit{Validation and Implementation Scope.}
The validation component was extended to enforce both chaincode-level and collection-level endorsement policies, and to perform the blind OCC validation described in \S\ref{s:design} against the hashed namespace in the State DB. The private data collection feature comprises approximately 33,000 lines of Go code, including production logic and tests.
