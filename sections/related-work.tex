\section{Related Work}

\textit{Channels and Sub-Networks.}
Hyperledger Fabric's \textit{channels}~\cite{hf} provide data isolation via independent blockchain instances, each with its own ledger and consensus. While effective, channels are coarse-grained: each data-sharing relationship may require a separate channel, leading to operational complexity. Our mechanism provides fine-grained privacy within one channel.

\textit{Private Transactions via Off-Chain Managers.}
Quorum~\cite{Quorum} and Hyperledger Besu~\cite{besu} implement a hash-on-chain / data-off-chain model using an external transaction manager (Tessera), but both have static privacy group membership, require all participants to be online, and offer limited interoperability between private and public state. Our mechanism supports dynamic membership with retroactive reconciliation, tolerates offline peers via gossip, and enables atomic cross-state reads and writes. R3 Corda~\cite{corda} eliminates the global ledger entirely, sharing transactions only on a need-to-know basis, but this introduces a \textit{backchain resolution} problem~\cite{corda}: verifying an input state requires walking prior transactions that may involve unrelated parties, inadvertently exposing historical data. This also precludes network-wide queries and the tamper-evident audit trail that on-chain hashes provide.

\textit{Cryptographic Privacy.}
Zero-knowledge proof systems provide strong privacy guarantees: Zerocash~\cite{inproceedings} uses zk-SNARKs~\cite{10.5555/2671225.2671275} for fully anonymous payments; Hawk~\cite{7546538} extends this to smart contracts but requires a trusted manager; Zether~\cite{Bnz2019ZetherTP} enables confidential token transfers via ElGamal encryption and range proofs. All incur significant computational cost (seconds per proof), making them unsuitable for high-throughput enterprise workloads. Our work is complementary---MediLedger~\cite{mediledger-fda, mediledger-hf} demonstrates combining PDCs with ZKPs in practice. Confidential Transactions~\cite{maxwell} and CryptoNote~\cite{Saberhagen2013CryptoNoteV2} target permissionless cryptocurrency privacy, not selective access control in permissioned networks.

\textit{TEEs, MPC, and Homomorphic Encryption.}
TEE-based systems (Ekiden~\cite{ekiden}, CCF~\cite{ccf}) execute contracts inside hardware enclaves but introduce hardware dependencies and side-channel vulnerabilities. MPC platforms (Enigma~\cite{enigma}) enable joint computation over private inputs but incur high overhead. Fully homomorphic encryption~\cite{gentry-fhe} allows computation on encrypted data but costs $10^4$--$10^6\times$. Our approach sidesteps these costs: authorized peers process private data in plaintext and store only hashes.

\textit{Off-Chain State Anchoring.}
State channels (e.g., Lightning~\cite{lightning}) and rollups (Optimistic~\cite{optimistic-rollups}, ZK~\cite{zk-rollups}) anchor off-chain state on a base ledger but address scalability in permissionless networks, not privacy in permissioned ones. Both assume a public verification layer that any party can challenge, whereas PDCs restrict data visibility to authorized collection members and rely on blind validation rather than fraud or validity proofs.

\textit{Database-Layer Privacy.}
CryptDB~\cite{cryptdb} executes SQL over encrypted data with practical overhead (15--26\%) but progressively leaks information. Differential privacy~\cite{dwork-dp} provides statistical guarantees against individual record disclosure. These complement our work: PDCs control \textit{who} accesses data at the replication layer, while database techniques limit \textit{what} queries reveal.

\textit{Positioning.}
Our mechanism achieves fine-grained privacy through architecture---selective replication, on-chain hashes, and blind validation---rather than expensive cryptographic primitives or hardware trust assumptions. Each alternative sacrifices at least one property: channels forfeit granularity, removing the global ledger forfeits the audit trail, proofs forfeit throughput, and enclaves forfeit deployment flexibility. PDCs avoid these trade-offs by recording only hashes on the shared ledger.
